## Step 0：判断用户意图

- 用户提供设计稿/标签文件 → 进入审核流程
- 用户说"对比版本""看看改了哪里" → 进入版本比对流程
- 用户补充企标/供应商标签/客户规则 → 进入知识入库流程
- 🆕 用户上传原料/供应商标签说"看下这个原料" → 进入原料预审流程

### 文件格式优先级
审核准确度从高到低：
1. **AI/PDF 未转曲（有文字层）** → pymupdf 直接提取文字，100%准确
2. **AI/PDF 已转曲（无文字层）** → 渲染300dpi→重叠分块OCR（10%重叠，边界去重）
3. **高清PNG/JPG（>2500px）** → 百度OCR
4. **低分辨率图片（<2500px）** → 标记风险后OCR
5. **定稿文案** → 直接比对

> AI文件中条码有3种形态：文字数字（可提取）、字体图元（部分可读）、嵌入位图（不可提取）。条码不在文字层时回退OCR。

### 5. OCR配额保护 + 质量三道检测（审核前强制执行）

#### OCR调用铁律（不可违反）

| # | 规则 | 违反后果 |
|:--:|------|------|
| 1 | **默认缩图至3.9MP**，用 accurate_basic，1次/张 | 月配额仅1000次，分块消耗过快 |
| 2 | 仅当缩图OCR未通过三道质检 → 该单张回退原图分块 | 分块≈15次/张，仅用于必要场景 |
| 3 | 🔴 **分块数 > 50 → 立即中断**，报"分块数异常({n})，可能公式错误" | 防2026-06-02的330×2=660块bug复发 |
| 4 | 🔴 **禁止用 general_basic 作为审核数据源** | 精度不足，宁报"OCR失败"也不给错数据 |
| 5 | 🟡 审计前打印 `本次预计消耗{N}次，月剩余{M}次` | 余额<100时提醒用户 |
| 6 | 多文件时**先试第1张**验证OCR质量，确认策略后再批量 | 防批量失败浪费配额 |

#### 两级OCR策略

```
输入图片
    ↓
缩图至3.9MP → accurate_basic（1次）
    ↓
三道质检（检测1/2/3）
    ↓
┌─ 质检通过 → ✅ 进入审核流程
└─ 质检不通过 → 仅此张回退原图重叠分块OCR（~15次）
```

#### 原图分块OCR（仅回退触发）

仅在缩图OCR质检不通过时触发：
- 将图片切成多块（每块≤4MP，满足百度OCR限制）
- **块间10%重叠**，边界文字不会被截断
- 合并结果时相邻块重叠区去重
- 分块公式见 CLAUDE.md → OCR配额管理

#### 检测0（新增）：单图分块数异常侦测
分块计算后立即检查：
- 分块总数 ≤ 50 → ✅ 正常，继续OCR
- 分块总数 > 50 → 🔴 中断：\"分块数异常（{cols}×{rows}={total}），可能计算错误。原因为缩图公式错用或图片超大至>200MP。已拦截以避免配额浪费。请手动确认图片尺寸后重试。\"

#### 检测1：分辨率告警
图片加载后立即检查：
- 最长边 < 1500px → 🔴 拦截：\"图片分辨率过低（{尺寸}px），OCR不可靠。建议提供300dpi以上高清图或定稿文案。\"
- 最长边 1500-2500px → 🟡 提醒：\"分辨率一般，可能有遗漏。建议提供更高清版本。\" （继续执行但标记风险）
- 最长边 > 2500px → ✅ 通过

#### 检测2：关键字段完整性
OCR完成后检查必须存在的字段：
| 字段 | 检测方式 | 缺失时 |
|------|---------|--------|
| SC编号 | 正则 `SC\d{14}` | 🔴 \"SC编号未识别，OCR可能大量遗漏\" |
| 配料表引导词 | 关键词 `配料` | 🔴 \"配料表未识别\" |
| 营养成分表 | 关键词 `能量\|蛋白质\|脂肪\|NRV` | 🔴 \"营养成分表未识别\" |
| 净含量 | 正则 `\d+.*克\|g\|千克\|kg\|ml\|毫升` | 🟡 \"净含量未识别\" |
| 产品标准号 | 关键词 `GB\|SB\|Q/\|产品标准` | 🟡 \"产品标准号未识别\" |

- 缺2个及以上🔴字段 → 拦截：\"OCR疑似大范围遗漏，请提供定稿文案或高清图。\"
- 缺1个🔴字段 → 🟡 提醒继续审核但标记高风险

#### 检测3：配料表项数合理性
配料表按顿号/逗号分隔统计项数：
- < 5项 → 🟡 \"配料表仅识别{N}项，项数偏少，大概率OCR遗漏。建议提供定稿文案补充审核。\"
- 5-8项 → 正常，与同类产品比对确认
- > 8项 → ✅ 正常

#### 检测不通过的处理
- 分辨率不足 → 直接拦截，不继续审核
- 关键字段缺2+ → 直接拦截
- 配料项数偏少 → 继续审核但审核报告顶部标注⚠️\"OCR可能遗漏配料\"，同时提醒提供定稿文案



### 6. 双通道OCR交叉验证

设计稿常含正面清晰层 + 镜像/重复印刷层，两通道OCR结果可能不同。

**处理策略**：
- 识别到多个文字层 → 清晰层作为**主审核依据**，镜像层不做为判定基准
- 清晰层与镜像层交叉比对：
  - 一致 → ✅ 高置信度
  - 不一致 → 🟡 \"此处双通道OCR不一致，请肉眼确认\"，**绝不自动判错**
- 只有一个文字层 → 直接审核，标记\"单通道OCR\"
- 优先使用终稿文案，AI/PDF/JPG仅做版面检查

**芥末蟹味棒教训**：清晰层OCR=磷酸酯✅，镜像层OCR=磷酸脂。不应因镜像层乱码误报🔴。

### 7. 营养成分表历史交叉校验（弱提醒）

审核时提取营养5项数值，与同品类历史基准比对：

**触发条件**：
- 同品类（同工厂+同标准）历史样本 >= 20个
- 当前仅 DWT(SB/T 10379) 满足条件（47样本）

**校验逻辑**：
```
当前值 vs 同品类中位数 → 偏差率 = |当前值-中位数| / 中位数 × 100%
  → 偏差率 <= 50% → ✅ 正常
  → 偏差率 > 50% → 🟡 "营养值{项}与同类产品中位数偏差{N}%，请确认"
                    （仅为提醒，不判为错误）
```

**输出格式**（审核报告底部）：
```
📊 营养历史校验（DWT / SB/T 10379 / 47样本）：

| 营养素 | 当前值 | 同类中位数 | 偏差 | 判定 |
|--------|:--:|:--:|:--:|:--:|
| 能量 | {x}kJ | {m}kJ | {d}% | ✅/🟡 |
| 蛋白质 | {x}g | {m}g | {d}% | ✅/🟡 |
| ... | ... | ... | ... | ... |
```

**不触发的情况**：
- 样本<20 → 提示 "同类产品样本不足（{N}/20），暂不校验"
- 不同品类 → 不交叉对比（水产和烘焙营养素结构完全不同）

### 8. 多文件去重合并

用户可能同时提供AI/PDF + JPG/PNG等多份文件：
- 先逐份提取三要素（产品名+客户+规格）
- 三要素匹配 → 同一产品，**合并处理**：AI/PDF用于文字审核（最准），JPG用作标注稿底图，**只输出一份审核报告**
- 三要素不匹配 → 不同产品，各出各的报告
- 只给AI/PDF → pymupdf渲染为PNG做标注稿底图

### 9. 一次完成原则

## Step 1：确认产品身份三要素

从设计稿/文件名/OCR 中提取：**产品名称 + 客户 + 规格（净含量）**。
三要素共同构成版本链唯一标识。

## Step 2：版本链匹配

搜索版本链目录，按三要素匹配：
- 精确匹配 → 关联为 V(N+1)，自动生成与上一版的差异报告
- 无匹配 → 新建 V1
- 模糊匹配（客户相同、名称相近、规格不同）→ 提示用户确认

## Step 3：执行审核

按审核清单逐项检查（见下）。

审核结果处理规则：
- P0/P1 级问题：先向用户报告问题概要，询问"输出审计稿还是先修改后重新提交再出稿？"
  以用户答复为准。
- P2 级问题：直接输出标注稿，标记存疑供人工确认。
- 定稿文案与设计稿内容不一致：标记为 🔴高优先级 ，在报告中单独列出差异供审核人确认。
- **条码提取+归属校验**：
  1. OCR后用正则 `69\d{11}` 提取EAN-13条码，还原13位，做格式校验（13位/69开头/校验位）
  2. 🆕 **校验位自动计算**（EAN-13算法）：
     ```
     前12位奇数位×1 + 偶数位×3（从左数第1位×1,第2位×3,第3位×1,...第12位×3）
     → 总和÷10余数R → 校验位=(10-R)%10
     → vs OCR读取的第13位 → 一致✅ / 不一致→🟡"校验位不匹配，OCR可能误读"
     ```
  3. 提取前7-9位厂商识别码，与工厂号段清单比对：
  
  | 厂商识别码 | 工厂 | 来源 |
  |:--:|------|------|
  | 697170406 | DWT 鼎味泰 | 条码证书 ✅ |
  | 697546729 | LYG 莱卡尔海洋 | 条码证书 ✅ |
  | 697901248 | LYGHB 莱卡尔烘焙 | 条码证书 ✅ |
  | 697160788 | LKE 莱卡尔太仓 | 条码证书 ✅ |
  
  4. **条码归属 vs 委托关系** 交叉校验（核心规则）：
  ```
  条码在工厂自身号段内 → ✅ 自有品牌，无需委托方
  条码不在工厂自身号段内：
    ├── 标签有"委托方"/"委托商" → ✅ 代工产品，条码属客户
    └── 标签无"委托方"/"委托商" → 🔴 "条码不在工厂号段且未标示委托方"
  ```
  5. 条码不在已知号段但有委托方 → 🟡 "条码不在已知号段，疑为客户号段，请确认"
### 重复问题防疲劳机制

同批次内多份标签有同一问题 → 合并为一条提醒，不逐份重复。

复合配料未展开时，附带供应商库提示帮助快速确认：
```
复合调味料未展开 → 🟡 +
  "供应商库：泰州贝特尔·GB 31644（已收录）
   → 是否为此供应商？如是可豁免展开。如否，请提供新供应商标签。"
```

不自动推测供应商，不跨批次记忆豁免状态。用户回复确认后才标记。

5. 条码审核报告中展示格式（审核报告统一使用此模板）：
  ```
  条码：697xxxxxxxxxxxx
  ├── 格式：✅/❌ 13位/69开头
  ├── 校验位：✅ 自洽 / 🟡 不匹配(计算=X, 读取=Y)
  ├── 厂商码：✅ 匹配 {工厂名} / 🟡 未匹配，疑为客户号段 / 🔴 不匹配且无委托方
  └── 注册状态：⚠ 未联网核验注册状态及产品真实性
  ```
  6. ⚠ 低分辨率图片（<2500px）条码OCR可能误读，此时条码异常标记🟡而非🔴

### 🚧 条码审核当前限制

| 限制 | 说明 |
|------|------|
| 无法解码条码图案 | 设计稿条码占比过小/常为占位图，zbar/zxing/OpenCV均无法可靠解码 |
| 依赖OCR读数字 | 条码下方人眼可读数字，OCR提取后做格式+厂商码校验 |
| AI条码不可提取 | AI文件内条码常为嵌入位图，文字层提取不到 |
| 不联网查注册 | 编码中心无公开API |
| 低分辨率误读 | <2500px时OCR可能读错数字，不判🔴 |
| 环境限制 | 条码解码库需系统DLL/Java，云端不具备。本地PC装zbar后可实现 |

每次审核完成后，按以下三层执行知识库更新：

### ✅ 静默自动（审核完直接做，不打扰用户）

### 🟡 审核报告末尾附加知识库更新摘要

每份审核报告底部增加摘要区块（不阻塞流程）：
```
📚 本次知识库自动更新：
  + 配料词库新增 N种：{名称}（🆕待确认）
  + 频次升级：{名称} N→N+1次
  + 供应商标签入库 N份：{名称}
  → 已自动入库。如需修正，随时告知。
```

### ⚠️ 新规则入库前三步校验（不可跳过）

**任何新规则/新错字/新阈值入库前，必须：**
1. **法规溯源** → 引用具体 GB 条款号，不能靠常识记忆
2. **逐条验算** → 数值或清单类规则必须逐条对照法规原文验算，发现任何一条不合规 → 整个方案废弃
3. **历史回测** → 用最近 3 份已审核标签回测，误报率>0 → 不入库

> 白名单事故(2026-06-12)：提出的 13 项"合法短配料白名单"7 项不合规，如未纠正将导致系统性漏判。
> 两条底线：①审核时宁可多报不可漏（可疑标🟡）②写知识库时不确定不入库（入必准确）。假规则比单次漏判严重百倍。

```

【审核引擎】
├── 规则引擎：58条规则(A~L类) + 334错别字 + 四工厂条码校验
```

# 审核清单（15步，不可跳过）

## 1. 基本信息校验
- SC编号：SC+14位数字，与生产主体库匹配
- 🆕 **SC编号食品类别交叉校验**：提取SC号→匹配生产主体库→提取该主体许可食品类别→与标注的产品标准号/类别比对：
  ```
  SC编号 → 生产主体 → 许可类别清单
    ├── 当前产品标准号在许可清单中 → ✅
    ├── 不在许可清单中 → 🔴 "SC号{编号}对应的{主体}许可范围不含{标准号}，疑超范围生产"
    └── SC编号不在任何已知主体 → 🔴 "SC编号{编号}未匹配任何生产主体，疑伪造或录入错误"
  ```
  示例：SC11132072201792=LYG(速冻食品+水产制品) → 产品标GB 10136→✅水产制品类别匹配
  示例：SC12432072202186=LYGHB(糕点) → 产品标GB 10136→🔴糕点企业不可生产水产制品
- 产品标准号与产品类型匹配
  GB 7099=强制安全标准（面包+糕点全品类必标）
  GB/T 20981=推荐品质标准（仅面包可选标，糕点不适用）
  SB/T 10379=速冻调制食品 / GB 10136=水产制品
  ⚠ GB 7099 + 起酥面包/调理面包 = 完全合规（之前我理解错了）
- 生产商名称+地址+产地完整
- 🆕 **委托加工信息完整性**：有委托方时须检查：
  | 检查项 | 缺失时 |
  |------|:--:|
  | 委托方名称+地址+联系方式 | 🔴 |
  | 受委托方SC号+产地 | 🔴 |
  | 产地=受委托方实际生产地址（地级市级别一致） | 🟡 |
  | 条码归属=委托方（非受委托方自身号段） | 见条码校验 |

## 2. 产品名称合规
- 真实属性 + 特征配料在配料表中可找到 + 无禁用词
- 🆕 **产品名称实质一致性**：名称中有实物配料→配料表中该实物应在香精/调味料之前出现：
  ```
  "芝士贝果" → 配料表中"芝士/干酪"应在"芝士味调味粉"之前 → 否则🟡"实物配料疑为香精调味"
  "蓝莓XX" → 配料表中"蓝莓"应在"蓝莓香精"之前 → 否则🟡
  "鳕鱼蟹柳" → 配料中应有真实鳕鱼糜(非混合鱼糜) → 否则🟡"名称含鳕鱼但配料为混合鱼糜"
  ```
  > 原则：名称暗示的实物配料必须真实存在且在香精/调味料前。香精调味时应用"XX味"命名。
- 🆕 **"味"字使用合规**：
  ```
  名称含"味"字（蟹味柳/牛肉味XX）：
    ├── 配料中有对应香精/调味料 → ✅ 合规（风味来自香精）
    ├── 既无实物也无香精 → 🟡 "风味来源不明，疑虚假宣称"
    └── 名称无"味"字但仅有香精无实物 → 🔴 误导（"蟹肉柳"但仅蟹味香精=违规）
  ```

## 3. 配料表合规
- 引导词/展开规则/添加剂标准名/排列顺序/客户禁限用
- 🆕 **前三推定**：配料表前三（水除外）默认推定可能≥25%，未展开时主动确认（K01规则）
- 🆕 **供应商标签匹配**：比对已有供应商标签，缺失的提醒用户补充（K02规则）
- LKE面包用小麦粉展开基准：(小麦粉、谷朊粉、硬脂酰乳酸钙、维生素C)
- LKE奶油制品展开基准：(稀奶油、奶油、生乳/牛生乳)
- 🆕 **特殊添加剂附带声明**：
  | 添加剂 | 必须附带声明 | 缺失 |
  |------|------|:--:|
  | 阿斯巴甜 | "含苯丙氨酸" | 🔴 GB 2760强制 |
  | 二氧化硫/亚硫酸盐(≥10mg/kg) | 须在配料表中标注 | 🔴 GB 7718强制 |
  | 糖醇类（山梨糖醇/麦芽糖醇/木糖醇等） | 建议标注"过量食用可能引起腹泻" | 🟡 GB 28050建议 |
  | 氢化油/部分氢化油 | 营养表须标反式脂肪 | 见Step 4 |
- 🆕 **辐照/转基因原料标示**：
  | 情形 | 要求 | 缺失 |
  |------|------|:--:|
  | 任何配料经辐照处理 | 配料表标"辐照XX" | 🔴 |
  | 使用转基因大豆/玉米等原料 | 须标示转基因信息 | 🔴 (瑞幸额外禁止) |
  > 当前鱼糜板块(DWT/LYG)不涉及；烘焙板块(LKE/LYGHB)的大豆油/玉米淀粉可能涉及→命中相关配料时主动提醒。
- 🆕 **单字食材检测**（配料名称未反映真实属性）：
  | 错误写法 | 正确写法 | 原因 |
  |------|------|------|
  | 鸡 | 鸡肉/鸡胸肉 | GB 7718-2025 §4.2.1 配料名称应反映真实属性 |
  | 猪 | 猪肉/猪瘦肉 | 同上 |
  | 牛 | 牛肉 | 同上 |
  | 羊 | 羊肉 | 同上 |
  | 鸭 | 鸭肉 | 同上 |
  | 鱼（非特指鱼糜制品） | 鱼肉/具体鱼种 | 同上 |
  | 虾 | 虾肉/虾仁 | 同上 |
  | 蛋 | 鸡蛋/全蛋液 | 同上 |
  > **检测逻辑（修正版）**：将配料表按分隔符（`、`, `,`, `)`, `(`）拆分为独立片段 → 找仅1-2个汉字且两侧为分隔符的孤立片段 → **排除"水"**（唯一合规的单字配料名）→ 其余全部 🔴
  > ⚠️ 白名单方案已废弃。经验证：盐/糖/油/醋/酱/蜜/茶/奶/酒等均不可单字使用（须写"食用盐""白砂糖""大豆油"等）。中国法规体系下，**除"水"外无一合法单字配料名**。此规则简洁准确，100%无漏判。
  > 典型案例：火腿片展开配料中仅写"鸡"→应为"鸡肉"；仅写"猪"→应为"猪肉"；仅写"盐"→应为"食用盐"。

## 4. 营养成分表合规
- NRV%计算/0界限值/单位/修约/5项齐全/反式脂肪/声称一致
- 🆕 **能量一致性校验（品质要求，严于国标）**：计算值=蛋白×17+脂肪×37+碳水×17，与标示能量比对，不一致即🟡"虽在国标允差内但需确认"；注意单位勿漏"千"
- 🆕 **反式脂肪触发条件检查**：配料命中`氢化|起酥油|人造黄油|植脂末|代可可脂|人造奶油|部分氢化`→营养表必须有反式脂肪酸行→缺失🔴"使用了氢化相关配料但未标示反式脂肪酸，违反GB 28050"
- 🆕 **"0"界限值自动反查**：营养值低于0界限值却标示为非0→🟡"应修约为0"：
  | 营养素 | 0界限值(/100g) |
  |------|:--:|
  | 能量 | ≤17kJ |
  | 蛋白质 | ≤0.5g |
  | 脂肪 | ≤0.5g |
  | 碳水化合物 | ≤0.5g |
  | 钠 | ≤5mg |
  | 反式脂肪 | ≤0.3g |
  > 示例：脂肪标0.3g→应标0；钠标3mg→应标0。此项常被忽略但监管抽检必查。
- 🆕 **营养成分表格式规范**（GB 28050表1）：
  | 检查项 | 异常 |
  |------|:--:|
  | 1+4齐全（能量+蛋白+脂肪+碳水+钠） | 缺→🔴 |
  | NRV%列存在 | 缺→🔴 |
  | 单位正确（kJ非KJ，g非G，mg非MG） | 错→🟡 |
  | 表头"每100克(g)"或"每100毫升(mL)" | 缺→🟡 |
  | 修约间隔（能量1kJ/蛋白0.1g/脂肪0.1g/碳水0.1g/钠1mg）| 不符→🟡 |

### 🆕 AI未转曲文件：字体大小精确检查

当输入为**AI未转曲文件**（有文字层）时，pymupdf可提取每个文字元素的 **pt值** 和 **坐标位置**，实现远超OCR精度的字体检查。

**换算公式**：`mm = pt × 0.3528`（1pt = 0.3528mm，72pt = 25.4mm = 1英寸）

**按区域划分的字高底线**：

| 标签区域 | 法规底线(mm) | 对应最小pt | 依据 |
|------|:--:|:--:|------|
| 净含量 ≤50g | 2mm | 5.7pt | GB 7718 表3 |
| 净含量 50-200g | 3mm | 8.5pt | GB 7718 表3 |
| 净含量 200-1000g | 4mm | 11.3pt | GB 7718 表3 |
| 净含量 >1000g | 6mm | 17.0pt | GB 7718 表3 |
| 品名/配料/营养/SC/日期/产地等 | 1.8mm | 5.1pt | GB 7718 3.9 |
| 自愿性声明（温馨提示等） | 1.8mm | 5.1pt | GB 7718 3.9（不得小于强制内容） |

**批量检查流程**：
```
1. pymupdf提取全部文字 span：{text, pt, bbox(x0,y0,x1,y1)}
2. 按 bbox.y0 从上到下分区：
   ├── 正面上方 → 品名区
   ├── 配料表 → 配料区
   ├── 营养成分表 → 营养区
   ├── 生产信息 → 强制标示区
   └── 温馨提示 → 自愿声明区
3. 逐区读取最小pt值，与底线比对：
   → ≥底线 → ✅
   → <底线 但 ≥底线×80% → 🟡"XX区最小字号{pt}pt({mm}mm)，接近底线{底线}mm"
   → <底线×80% → 🔴"XX区最小字号{pt}pt({mm}mm)，低于法定底线{底线}mm"
4. 输出字体检查报告（仅列出接近/低于底线的区域，全部通过则显示✅）
```

**输出示例**：
```

| 区域 | 最小字号 | 换算mm | 底线 | 判定 |
|------|:--:|:--:|:--:|:--:|
| 品名区 | 7.2pt | 2.5mm | 1.8mm | ✅ |
| 配料区 | 5.5pt | 1.9mm | 1.8mm | ✅ |
| 营养表区 | 4.8pt | 1.7mm | 1.8mm | 🟡 接近底线 |
| SC/日期区 | 5.0pt | 1.8mm | 1.8mm | ✅ 刚好达标 |
```

> 📌 **仅AI未转曲文件执行此精确检查**。图片/JPG/PDF渲染文件回退到OCR坐标框估算，标记🟡提醒人工确认。
> 📌 净含量字高根据实际克数选用对应底线行；其余区域统一用1.8mm底线。

### 🆕 语病/语义通顺检查

标签中常见的语句不通、歧义、成分残缺，集中在以下区域：

#### 致敏原区域（最高发）

| 语病类型 | 检测方式 | 示例 |
|------|------|:--:|
| **句子粘连** | 正则 `制品.*本生`（制品后无句号直接接"本生"） | "含大豆制品本生产线也生产"→应为"含大豆制品。本生产线也生产" |
| **交叉污染句成分残缺** | "本生产线"+"也" 但无 "可能" → 缺修饰词 | "本生产线也生产"→应为"本生产线**也可能**生产" |
| **致敏原列举后缺句号** | 致敏原段以 `。` 结尾 → 检查是否缺句号 | 整段致敏原无句号 → 补充 |

#### 配料表区域

| 语病类型 | 检测方式 | 示例 |
|------|------|:--:|
| **配料名粘连** | 两个中文词之间无顿号/逗号/括号分隔 → 可能漏分隔符 | "白砂糖饮用水"→应标"白砂糖、饮用水" |
| **重复"的"字** | 正则 `的的` | "麸质的谷物"→正常；"的的"连续→🟡 |
| **"等"字疑似暗示遗漏** | 配料表末尾出现"等" → 🔴 违规（已有Step 3覆盖） | — |

#### 贮存/食用方法区域

| 语病类型 | 检测方式 | 示例 |
|------|------|:--:|
| **条件句中缺"后"** | "解冻"+"食用"中间无"后/即可" | "解冻食用"→应为"解冻**后**食用"或"解冻**即可**食用" |
| **温度与动作断句不清** | 数字+"°C"+"解冻"，中间无空格或标点 | "0-8°C解冻后"→建议"0-8°C**冷藏**解冻后" |
| **"请勿再次冷冻"缺主语** | 单独出现"请勿"但无语境提示 | "解冻后请勿再次冷冻"→已完整 ✅ |

#### 通用区域（全标签扫描）

| 语病类型 | 检测方式 | 示例 |
|------|------|:--:|
| **"的"字连续 3 次以上** | 正则 `的.{0,5}的.{0,5}的` | "麸质的谷物的制品的产品"→🟡 累赘 |
| **中英文括号混用** | `（` vs `(` 同文出现 | 配料表中英文括号混用→🟡 |
| **冒号后连写无空格** | 中文冒号后紧接文字（正常）→排除；英文冒号后无空格→🟡 | "贮存条件：-18℃" ✅；"贮存条件:-18℃"→🟡 |

> 📝 语病类问题全部标记 🟡 提醒人工确认。仅当语病导致**实质性歧义**（如即食vs非即食混淆）时才升级为 🔴。
> ⚠️ 本条规则为新增（2026-06-14），基础版本覆盖致敏原/配料/贮存三区域高频语病，后续随审核积累持续扩展。

提取标签中所有非配料/非营养/非法规的文本，逐条检查：

| 检查项 | 规则 |
|--------|------|
| 语义断裂 | 宣传语明显不完整→🟡"此处宣传语疑似缺失，请确认" |
| 无支撑宣称 | 声称"含/高/低/富含/无"XX → 查验配料或营养表 → 不达标→🔴。阈值速查见下表 |
| 缺免责声明 | 有食物照片但无"图片仅供参考"/"图案仅供口味参考"→🟡 |
| 缺强制提示 | 营养成分表后缺"儿童青少年应避免过量摄入盐油糖"→🟡 |
| 禁用词 | 错别字库第七类 + 广告法第9条逐词匹配→命中即🔴 |
| 🆕 暗示疾病预防/治疗 | "增强免疫力""降血糖""抗衰老"等→🔴违反食安法第73条 |
| 🆕 贬低其他食品 | "比XX更好""非XX可比"等→🟡 |
| 🆕 自愿声明与强制内容矛盾 | 温馨提示暗示的食用/贮存方式与标示不一致→🔴 |
| 🆕 自愿声明字号 | 温馨提示等文字不得小于强制标示内容→OCR估算后接近临界值→🟡 |
| 🆕 "零添加"/"无添加"声称 | 须配料全链追溯（含复合配料带入）。如"零添加防腐剂"→复合配料酱油含苯甲酸钠→🔴 | 检测逻辑见下方专项 |
| 🆕 声称营养素不在营养表中 | 声称"高钙/富含膳食纤维/含DHA/低胆固醇"等→该营养素必须在营养成分表中有独立行→缺失🔴 | 违反GB 28050 4.3 |
| 🆕 免责声明字号 | "图片仅供参考"等字样须≥1.8mm（属强制标示内容的一部分）→AI未转曲文件直接读pt验证→不达标🔴 | 参考Step 11字体检查 |

### 🆕 "零添加"/"无添加"声称专项检查

打假人最高频下手点。检测逻辑分三层：

**第一层：关键词命中**
```
标签出现"零添加""无添加""0添加""不含添加""不添加"→触发此检查
```

**第二层：声称对象匹配**
```
"零添加蔗糖" → 配料表含果葡糖浆/蜂蜜/浓缩果汁→🔴"虽未添加蔗糖但含果糖/葡萄糖，零添加声称误导"
"零添加防腐剂" → 配料表或复合配料含苯甲酸钠/山梨酸钾/脱氢乙酸等→🔴
"零添加色素" → 配料表或复合配料含辣椒红/焦糖色/胭脂虫红等→🔴
"零添加香精" → 配料表含食用香精→🔴
"无添加糖" → 配料含白砂糖/葡萄糖浆/麦芽糖浆→🔴
```

**第三层：复合配料带入追溯（关键）**
```
声称"零添加XX" 
→ 遍历配料表中所有复合配料（有括号展开的、酱油/调味料/酱料等已知复合的）
→ 检查复合配料的展开项中是否含声称对象
→ 含→🔴"复合配料{名称}中含有{XX}，'零添加{XX}'声称不成立"
```
> 示例：标签"零添加防腐剂"，配料含"酱油（水、大豆、小麦粉、食盐、苯甲酸钠）"→酱油带入苯甲酸钠→🔴

**输出格式**：
```
🔍 零添加声称专项检查：
声称："零添加蔗糖"
├── 配料直接检查：✅ 无白砂糖/蔗糖
├── 复合配料追溯：🔴 含"本味啉调味料（水、糯米、食用葡萄糖...）"→带入葡萄糖
└── 判定：🔴 "零添加蔗糖"声称可疑，复合配料带入糖类成分
```

> 打假人策略：不争论定义，直接主张"'零添加'字样误导消费者以为该产品优于同类"，且《食品标识监督管理办法》规定零添加声称需同时标注该成分为零的含量。标签上"零添加"字样本身就是打假人重点关注对象。

### 营养声称阈值速查（GB 28050-2025）

审核时提取标签上的声称关键词，逐条对照营养表验证：

| 声称 | 条件（每100g固体） | 不达标 |
|------|------|:--:|
| 低脂肪 | 脂肪 ≤ 3g | 🔴 |
| 无脂肪/零脂肪 | 脂肪 ≤ 0.5g | 🔴 |
| 低糖 | 糖 ≤ 5g | 🔴 |
| 无糖 | 糖 ≤ 0.5g | 🔴 |
| 高蛋白/富含蛋白质 | 蛋白质 ≥ 12g（≥20%NRV） | 🔴 |
| 高钙/富含钙 | 钙 ≥ 240mg（≥30%NRV） | 🔴 |
| 低钠 | 钠 ≤ 120mg | 🔴 |
| 高膳食纤维/富含膳食纤维 | 膳食纤维 ≥ 6g | 🔴 |
| 低能量 | 能量 ≤ 170kJ | 🔴 |
| 不含反式脂肪酸 | 反式脂肪酸 ≤ 0.3g | 🔴 |

> 本次更新来源：鳕鱼大蟹柳标签写"富含"但蛋白10.7g未达12g阈值→发现第12步缺少具体数值对照。衍生检查：标签上有任何声称关键词时，必须逐条对照此表验算。

### 13. 配料表排列顺序逻辑预判

审核时根据产品类别预设主料预期，与配料表前三比对：

| 产品类别 | 预期第一主料 | 异常模式 | 提醒 |
|------|------|------|:--:|
| 面包/贝果/吐司/可颂 | 小麦粉/面包用小麦粉 | 小麦粉不在前三 | 🔴"面包类产品主料非小麦粉，疑配料顺序错误" |
| 鱼糜制品（蟹柳/鱼丸/竹轮） | 鱼糜/鳕鱼糜/冷冻鱼糜 | 水排第一+鱼糜排第二 | ✅ 水产制品常见 |
| 鱼糜制品 | 鱼糜/鳕鱼糜 | 非水产肉类排前三 | 🟡"鱼糜制品中非水产配料占比异常" |
| 蛋糕/糕点 | 小麦粉+鸡蛋 | 小麦粉不在前三 | 🟡 |
| 芝士/奶酪命名产品 | 芝士/干酪 | 芝士排第5位以后 | 🟡"名称含芝士但配料排序靠后，疑含量偏低" |
| 水果命名产品 | 对应水果 | 水果不在配料表中或排香精后面 | 🔴 |

> 原则：产品类别决定主料预期。配料排列顺序必须符合递减规则。仅为逻辑预判，最终以实际投料量为准。

### 14. 配料声称与营养声称区分检查

标签上的声称须区分两类，各自对照不同依据验证：

| 声称类型 | 示例 | 核查依据 | 不达标 |
|------|------|------|:--:|
| 营养声称 | "低脂""高蛋白""富含钙" | 营养成分表→Step 12阈值表 | 🔴 |
| 配料声称 | "含真实蟹肉""添加蓝莓""鳕鱼制作" | 配料表→实物存在且在香精前→Step 2 | 🔴 |
| 含量声称 | "蟹肉≥15%" | 配料表→是否有定量标示 | 🟡"声称含量≥X%但未在配料表中定量标示" |
| 工艺声称 | "慢发酵""手工""现烤" | 暂无自动化验证 | 标记供人工确认 |

> 常见易混：标签写"富含鳕鱼糜"→"富含"是营养声称词→应查营养表蛋白质是否≥12g；而非仅查配料表有无鳕鱼。营养声称词+配料名=营养声称，必须达标。

### 15. 条码校验位自动计算

EAN-13校验位算法，用于验证OCR读取的条码数字是否自洽：

```
1. 取前12位数字
2. 奇数位×1 + 偶数位×3（从右往左数，含校验位的第1位=奇数位...即从左边数第1位×1,第2位×3,...）
   等效公式：∑(第i位 × (i为奇数?1:3))，i=1..12
3. 总和÷10，取余数
4. 校验位 = (10 - 余数) % 10
5. 计算出的校验位 vs OCR读取的第13位 → 一致✅ / 不一致→🟡"条码校验位不匹配，OCR可能误读某位数字"
```

示例：697546729084?
- 6×1+9×3+7×1+5×3+4×1+6×3+7×1+2×3+9×1+0×3+8×1+4×3 = 6+27+7+15+4+18+7+6+9+0+8+12 = 119
- 119÷10=11余9 → 校验位=10-9=1
- 实际第13位=1 → ✅ 校验通过

> 仅做OCR质量验证，校验失败的条码仍不联网查注册状态。

# 原料标签预审（5步轻量检查）

> 触发：用户上传原料/供应商标签，说"看下这个原料""这个能用吗""有没有问题"。
> 目的：在成品审核前提前扫清供应商端隐患，审核员审成品时已有答案。

## 与成品审核的关系

```
原料标签预审（5步）            成品标签审核（15步）
     ↓                              ↓
  发现问题 → 供应商整改          发现问题 → 设计师整改
     ↓                              ↓
  入库"供应商标签库"             入库"版本链"
     ↓                              ↓
  成品审核时自动匹配供应商库 → 配料展开规则已有答案
```

> 两套流程独立运行，知识库共享。原料预审的结果存入供应商标签库，成品审核 Step 3 的"供应商标签匹配"自动调取。

## 预审清单（5步）

### 1. 品名 + 标准号 + SC 三位一体校验

| 检查项 | 方式 | 异常判定 |
|------|------|:--:|
| 品名是否有国标支持 | 品名 vs GB/行标名称对照 | 名称模糊→🟡 |
| 执行标准是否与品名匹配 | 标准号→标准名称→产品类型 vs 品名 | 不匹配→🔴 |
| SC 编号食品类别是否覆盖该产品 | SC号→生产许可明细 vs 产品类型 | 不覆盖→🔴 |
| SC 格式（SC+14位数字） | 正则 | 缺位→🔴 |

> 示例：品名"香草调味酱"+ 标准 GB 31644（复合调味料）+ SC103（调味品）→ ✅ 三位一体匹配。

### 2. 配料表模糊用语扫描

| 关键词 | 可疑点 | 判定 |
|------|------|:--:|
| "提取物" | 含溶剂/载体未标？ | 🟡 追问供应商完整组成 |
| "XX制品""XX酱""XX膏" | 复合配料二级组成不明 | 🟡 供应商能提供才算过 |
| "食用油脂制品" | 类别名非具体名 | 🔴 |
| "等"字 | 暗示遗漏配料 | 🔴 |
| 配料 < 3 项但产品为酱/膏/复合形态 | 配料过少，疑有隐形成分 | 🟡 |

### 3. 常识推断——该有却没标的成分

| 产品形态 | 该有但配料表无 | 判定 |
|------|------|:--:|
| 酱/膏/液体状 | 无"水" | 🟡 含水量来源不明（糖浆自带？还是漏标？） |
| "提取物" | 无溶剂/载体说明 | 🟡 追问提取工艺和残留 |
| 标"调味酱"但仅1种调味料 | 复合调味料应≥2种调味料 | 🟡 "名称与实物实质不符" |
| 常温12个月保质期但无防腐剂 | 配方天然抑菌还是漏标？ | 🟡 标记供供应商确认 |

> 此项仅推断，不判违规。全部标记🟡供人工确认。

### 4. 添加剂 vs 成品工厂/客户禁区

- 逐项比对原料配料中的添加剂 → GB 2760 许可范围
- 交叉比对成品对应的客户禁用清单（瑞幸/ALDI 等）
- 添加剂含客户禁用的 → 🔴 "该原料含{XX}，成品用于{客户}时不可使用"

> 示例：原料含苯甲酸钠 + 成品供瑞幸 → 🔴 瑞幸禁用苯甲酸钠。

### 5. 用于成品时的展开规则预判

| 原料标准 | 加入量条件 | 成品配料表写法 | 判定 |
|------|:--:|------|:--:|
| 有国标/行标/地标 | <25% | 直接写原料品名，豁免展开 | ✅ |
| 有国标/行标/地标 | ≥25% | 须展开子配料 | 🟡 提醒采购确认用量 |
| 无国标/行标/地标 | 任何量 | 必须展开 | 🔴 必须展开 |
| 企标 | <25% | 须展开 | 🔴 企标不豁免展开 |

> 此项结果存入供应商标签库，成品审核 Step 3 自动调取。

# 审核输出格式

## 标注稿规范（横版A4 · 600dpi · 7016×4960px · 商务风格）

### 多文件审核优先级
同一产品多份文件时：
1. AI未转曲（有文字层）→ 文字100%准确，优先
2. AI已转曲（无文字层）→ 渲染300dpi→OCR
3. 最高清JPG/PNG → OCR
4. PDF → 渲染→OCR

标注稿底图：选最清晰那张。审核用最优文字源。只出一份报告。

### 布局规则
- **横版A4**：7016×4960px @600dpi，四周236px边距
- **左上角**：标注文件来源类型（AI转曲/AI未转曲/PNG/JPG/PDF）+ OCR时所用分辨率
- **左侧 65%**：原始设计稿，居中放置，浅灰边框，下方标注"包装稿件终版"
- **右侧 35%**：审核意见区，浅灰背景+边框包裹
- **中间**：浅灰色细线分隔
- **统一字体**：微软雅黑，无衬线，商务风格
- **不在设计稿上叠加标注**

### 右侧审核意见区（从上到下）
1. **标题**：AI 标签审核报告（蓝色加粗）
2. **副标题**：审核编号 + 时间（灰色小字）
3. **分隔线**
4. **产品信息**：产品名（大字号）+ 规格/工厂/标准（灰色）
5. **统计条**：白色卡片，🔴不通过 🟡存疑 ✅通过（横向排列，竖线分隔）
6. **不通过卡片**：白底+浅红标题栏，红色粗体条目
7. **存疑卡片**：白底+浅黄标题栏，黄色粗体条目
8. **通过卡片**：白底+浅绿标题栏，绿色条目
9. **NRV验证卡片**：白底浅绿，必须展示完整计算过程（如「444÷8400=5.29%→5%」），不可只写结果
10. **免责声明**：右下角，深灰色小字
11. **配料精简建议区块**（蓝色，💡参考色调）：每次最多2条，优先合并（多种淀粉→淀粉）再删字。标题含"仅供参考"，不判🔴🟡

| 用途 | 字号 |
|------|:--:|
| 大标题 | 60pt |
| 产品名 | 48pt |
| 条目正文 | 44pt |
| 条目加粗（不通过/存疑） | 40pt 微软雅黑粗体 |

# 概念解释

## 供应商标签映射表

供应商给你的原料标签名称通常很精确很长，你印在成品配料表上时会精简。映射表记录对应关系：

| 供应商原始名称 | 你的标签用名 | 是否复合配料 | 供应商配料展开 | 在成品中最大占比 | 备注 |
|------------|-----------|:--:|------------|:---:|------|
| 香满园800蛋糕专用小麦粉 | 小麦粉 | 否 | 小麦 | 15% | 安全 |
| 海天金标生抽酱油 | 酱油 | 是 | 水、黄豆、小麦粉、食盐、焦糖色、苯甲酸钠 | 30% | ⚠️>25%须展开 |
| 安佳大黄油 | 黄油 | 否 | 稀奶油 | 8% | 安全 |

**用途**：审核时自动检查你的精简是否跨过三条红线：
1. 复合配料 >25% 未展开
2. 简化后丢失关键信息（如添加剂带入）
3. 合并后名称有误导

## 反向索引

建立"被依赖方→依赖方"的快速查找能力，两个维度：

### 维度1：标准→产品
当国标/企标修订时（如 GB 7718 更新），一键查出所有引用该标准的标签，批量复核。

```
GB/T 20981 修订
  → 反向索引查出：
    全麦贝果_瑞幸_80g、岩烧牛乳吐司_美宜佳_85g、
    港式菠萝包_美宜佳_40g、... 共12个产品
  → 批量复核所有标签是否需要更新
```

### 维度2：原料→产品
当某批次原料出问题时（如某供应商小麦粉被召回），一键查出所有使用该原料的产品标签。

```
某批次小麦粉需要召回
  → 反向索引查出所有配料中含"小麦粉"的产品
  → 快速定位受影响标签，启动应急流程
```
